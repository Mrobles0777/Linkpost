🧩 Detalles y Vulnerabilidades Encontradas

Exposición de GEMINI_API_KEY en Clientes

🔴 Crítico

Archivo / Componente: vite.config.ts, src/services/geminiService.ts

Descripción del problema: La clave de la API de Gemini (GEMINI_API_KEY) se inyecta directamente en el bundle web, exponiendo la credencial maestra de facturación de Google Gen AI a cualquier usuario que inspeccione tu código.

Recomendación: Mueve toda la lógica de geminiService.ts al backend (server.ts) e invócala mediante una petición HTTP segura. NUNCA accedas a procesadores de lenguaje con token administrativo desde el front.

Inseguridad en el Payload (ID Spoofing)

🔴 Crítico

Archivo / Componente: server.ts (Ruta /api/linkedin/post)

Descripción del problema: El endpoint de publicación confía ciegamente en una variable (userId) que entra en el req.body extraído localmente. Así, cualquier persona que conozca el identificador de otro usuario, pude publicar un contenido en su favor por carecer de Auth Header o Session de Control.

Recomendación: El backend debiera requerir un header de "Authorization" (con un JWT), validarlo con Supabase para certificar que auth.uid() coincide con el de la solicitud, y utilizar este UID confiable, ignorando el del Body.

Acceso y RLS Supabase Abusado

🔴 Crítico

Archivo / Componente: server.ts y Base de Datos Supabase

Descripción del problema: El componente backend opera un .upsert() utilizando el SUPABASE_ANON_KEY (que no posee autoridad explícita de un UID). Si el Row Level Security de tu DB es óptimo, esta transacción fallará siempre. Si por error el sistema funciona, indicaría abiertamente que el RLS acepta intrusión anónima sin límites de lectura/escritura.

Recomendación: Actualiza server.ts utilizando variables como SUPABASE_SERVICE_ROLE_KEY de modo que funcione como Admin y logre evadir RLS global exclusivamente desde un ambiente validado por variables locales a nivel del backend.

Falta de Rate Limiting a Entradas API públicas

🟠 Medio

Archivo / Componente: server.ts (/api/image/search, /api/auth/linkedin/url)

Descripción del problema: Las API expuestas por GET en express no cuentan con validadores contra un uso masivo (spam y scraping / bots).

Recomendación: Instalar express-rate-limit configurado bajo una política restrictiva y justa en el app principal.
