🧩 DETALLES DE AUDITORÍA

CRÍTICO

Fuga de Claves API en Frontend

vite.config.ts | src/services/geminiService.ts

Descripción del problema: La clave secreta de Gemini (GEMINI_API_KEY) está siendo inyectada directamente en el lado del cliente (frontend) a través de los "defines" de un empaquetador moderno como Vite. Esto hace que las llamadas a la IA corran por la computadora del usuario y dejen la clave expuesta al código fuente visible en el explorador. Un actor malicioso puede robar la Key y sobrecargar la API generando fuertes perdidas de facturación.

💡 Recomendación de Antigravity:

Mueve toda la ejecución del SDK de Gemini a tu servidor (server.ts). El frontend solo debe realizar peticiones POST sin llaves al backend, siendo este el único garante capaz de contactarse bidireccionalmente y en entorno seguro (Server-side) con Google.

CRÍTICO

Bypass de Autorización (Insecure Direct Object Reference)

server.ts (Rutas /api/...)

Descripción del problema: Las rutas Express (`/api/linkedin/post`) toman acciones determinantes leyendo ciegamente el payload json userId para consultar tokens de un supuesto dueño de base de datos e invocar LinkedIn. No hay verificación mediante Token JWT autorizado ni Headers, por ende si cualquier sujeto externo hace un Request con el UUID de otra cuenta, podría emitir y forzar posteos a nombre de la víctima.

💡 Recomendación de Antigravity:

Elimina el parámetro estático userId y obliga a solicitar el token JWT nativo de la sesión en los Headers Authorization. Comprueba dicho token en el servidor mediante el cliente server-side supabase y extrae el user subyacente blindando por completo la interfaz REST.

MEDIO

Potencial Abuso de Reglas RLS

src/App.tsx

Descripción del problema: El cliente frontal usa operaciones .upsert() sobre las tablas (profiles). Si tu base de datos Supabase no tiene filtros restrictivos RLS predefinidos por rol, cualquiera podría corromper la DB o actualizar datos en ID falsificados desde la network console.

💡 Recomendación de Antigravity:

Constata exhaustivamente en el visor SQL remoto de tu Supabase Auth Settings que tus Row Level Security Policies prohíban modificaciones/inserciones que no concuerden bajo el requerimiento estricto WHERE auth.uid() = id.
